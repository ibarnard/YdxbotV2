# v2.0.1 Beta 交互闭环测试报告（2026-03-08）

## 1. 目的
这份报告记录 `codex/v2.0.1-beta` 在 2026-03-08 的一轮完整交互闭环测试结果。

目标不是只看“脚本能不能启动”，而是验证：

1. 管理员命令是否都能收到明确反馈。
2. 反馈是否统一回到指定 `admin_chat = 5721909476`。
3. 驾驶舱、值守摘要、告警、状态卡在 Telegram 手机端/桌面端是否足够清晰。
4. 当前代码行为是否和技术文档一致。
5. 现阶段是否具备继续做单账号小额实盘观察的条件。

## 4. 本轮已修复问题

### 4.1 命令反馈分散
问题：

- `model` / `apikey` 等子命令仍走 `event.reply(...)`
- 导致交互日志不完整，命令反馈语义不统一

处理：

- 统一改成命令类反馈走 `admin_chat`
- 新增 `_reply_admin_command_result(...)`

结果：

- `model list`
- `model reload`
- `model select 1`
- `apikey show`

都已经纳入统一出件路径和日志路径。

### 4.2 Windows 下 `ver` 命令崩溃
问题：

- `update_manager._run_cmd(...)` 使用系统默认编码
- Windows 本地读 git 输出时触发 `UnicodeDecodeError`
- 表现为 `ver` 最终回复：`命令执行出错: 'NoneType' object has no attribute 'strip'`

处理：

- `_run_cmd(...)` 显式改成 `encoding="utf-8", errors="replace"`

结果：

- `ver` 已恢复正常
- 真实链路返回了完整版本面板

### 4.3 帮助信息过长且语义过时
问题：

- 原 `help` 超过 3000 字，手机端阅读成本过高
- 旧文案仍写“`watch` 发到值守通道”，和当前实现不一致

处理：

- `help` 改成摘要版
- 新增 `help all`
- 更新 `watch` 的命令路由说明

结果：

- `help`：`662` 字 / `46` 行
- `help all`：`1881` 字 / `65` 行

默认帮助已明显更适合手机端查看。

### 4.4 `doctor` / `watch alerts` 长行不适合手机端
问题：

- 最近异常以前是一条超长单行
- 手机端阅读非常差

处理：

- 把最近异常拆成时间 / 阶段 / 错误类型 / 摘要三行

结果：

- `doctor` 最大单行长度从超 150 降到 `70`
- `watch alerts` 最大单行长度降到 `86`

### 4.5 任务与任务包总览命令提示过长
问题：

- `task` 单行命令提示过长，接近 180 字符
- `pkg` 也存在类似问题

处理：

- 拆成 `命令 / 详情 / 控制` 三行

结果：

- `task` 最大单行长度收敛到 `80`
- `pkg` 最大单行长度为 `56`

### 4.6 启动后旧 `startup` 异常一直挂在当前风险上
问题：

- 之前本地跨 IP session 异常虽然已经恢复
- 但 `doctor` 和 `watch alerts` 仍把它视为“当前风险”

处理：

- 新增 `clear_runtime_faults(...)`
- `start_user(...)` 成功启动后清理历史 `startup*` 异常

结果：

- 真实 round4 回归中：
  - `doctor` 不再展示旧 startup 故障
  - `watch alerts` 变成“暂无需要立即处理的事项”

### 4.7 资金不足暂停误报
问题：

- `fund_pause_notified` 残留时，`watch risk / watch funds / watch alerts` 可能误报“资金暂停”
- `tim` 当前资金实际够用，但仍显示“资金不足暂停”

处理：

- 新增 `_has_effective_fund_pause(...)`
- 只有资金实际小于当前所需下注额时，才继续显示资金暂停

结果：

- 真实 round4 回归中：
  - `watch risk` 显示 `资金暂停 否`
  - `watch funds` 显示 `资金暂停：否`
  - `watch alerts` 不再出现假阳性资金暂停

## 7. 与技术文档比对

### 7.1 已发现并修正文档偏差
这次测试前，文档与实际行为有 1 个明确偏差：

- `docs/tg-watch-v1-tech-spec.md`
- `docs/tg-watch-query-v1-tech-spec.md`
- `docs/system-technical-manual-v0.1.1.md`

原文更偏向“手动 `watch` 查询也走值守通道”。

当前真实行为是：

- 手动 `watch*` 查询统一回复 `admin_chat`
- 主动播报仍走 `watch` 通道

这三份文档已经同步更新。

### 7.2 仍然存在的文档边界
`docs/admin-cockpit-v1-tech-spec.md` 里描述的完整目标包括：

- 手卡
- 局卡
- 轮卡
- 日卡
- 告警卡

当前真正已经落地并通过本轮真实验证的是：

- 驾驶舱
- 手动暂停卡
- 手动恢复卡
- 部分模型/版本/状态类命令卡

所以这里不是“文档错误”，而是“文档写的是阶段目标，代码只落了其中一部分”。

## 8. 与 `main` 主线经验对照

对照 `origin/main` 当前 `zq_multiuser.py` 的命令入口和驾驶舱口径，这条 beta 分支当前已经补齐了几个关键点：

1. 启动后自动发送驾驶舱。
2. 命令类反馈统一回 `admin_chat`。
3. `model/apikey/watch/doctor/ver` 这些命令的回包路径不再分散。
4. 版本、值守、学习、自检视图已经形成比较完整的“管理面”。

当前仍未完全追平、或者说仍需继续打磨的地方：

1. 真正的每手下注卡 / 每手结算卡 / 局卡 / 轮卡 / 日卡还没有完整落地。
2. 真实朱雀消息驱动下的事件卡闭环，这次没有在真钱下注链路上完成验证。

---

## 9. 剩余问题与风险

### 9.1 当前仍建议继续优化的点
1. 自动暂停 / 自动恢复 / 跳过下注 / 观望 / 阻断 这些事件卡还没有完全统一。
2. `ver` 默认面板已经可用，但如果后续继续加信息，仍要警惕再次变长。
3. 驾驶舱和事件卡已经更适合手机端，但仍要以真实盘面滚动消息再验证一轮。

### 9.2 当前未覆盖的高风险区域
1. 真实 `bet_on -> 下单 -> settle` 的 Telegram 事件链路。
2. 真正带资金变化的 `resume/open/st` 实盘闭环。
3. `task run` / `pkg run` 对真实盘面的影响。
4. `update` / `reback` / `restart` 这些运维命令的真实执行链路。
5. `apikey set/add/del/test` 的真实写配置链路。

这些不是当前代码一定有问题，而是本轮测试为了不越界到真钱/重启/写密钥而刻意没有实操。

---

## 10. 结论
截至 2026-03-08，这条 beta 分支的“管理员命令交互 + 驾驶舱 + 值守查询 + 自检/版本面板”已经基本成型。

同一天内又做了一轮 UI/交互收口，已经继续完成了这些事：

1. 驾驶舱头部改成“账号 / 状态 / 策略 / 任务 / 资金 / 收益”摘要。
2. `help`、`learn`、`policy`、`ver`、`model` 面板继续压缩，手机端更容易扫读。
3. 暂停 / 恢复 / 下单 / 结算卡继续统一到“动作 / 当前 / 策略 / 原因 / 下一步”的表达。
4. 整组自动化回归提升到 `184 passed`。

如果只问一个问题：

> 现在能不能继续拿 `tim` 做单账号、小额、人工盯盘式 beta 观察？

答案是：

可以。

但如果问题变成：

> 现在能不能把“真实下注消息链路、事件卡、任务轮转、运维命令”也都视为完全验证完毕？

答案是：

还不能。

下一步最合理的工作不是继续加大功能面，而是按待办清单把“真实事件卡闭环”和“剩余高风险命令链路”做完。
