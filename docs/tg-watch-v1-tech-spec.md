# Telegram 值守播报 V1 技术说明书

## 1. 文档目的
这份说明书记录 `codex/risk-history-v1` 分支在 `I. Telegram 值守播报 V1` 阶段完成的能力。

目标不是做运营后台，而是把个人值守真正需要盯的关键信息稳定发到指定 `Telegram bot / chat`：

- 指定值守播报目标
- 主动播报关键事件
- 在 Telegram 内直接查看压缩后的值守摘要
- 提供聚合后的告警视图

---

## 2. 阶段定位
到这个阶段，系统已经具备：

- 单账号风控、动态押注、任务、任务包
- 策略版本化、灰度、回滚
- 受控自学习、影子验证、灰度转正回滚
- 多账号视图

缺的不是更多自动化，而是“我在指定 TG 通道里怎么快速看到关键状态”。

因此本阶段只解决一个问题：

> 让脚本把值守必需的信息用适合 Telegram 阅读的方式推出来。

## 4. 配置设计

### 4.1 统一配置入口
值守播报目标统一落到：

```json
{
  "notification": {
    "watch": {
      "admin_chat": -1001234567890,
      "tg_bot": {
        "enable": true,
        "bot_token": "xxx",
        "chat_id": "yyy"
      }
    }
  }
}
```

### 4.2 兼容旧字段
仍兼容历史写法：

- `notification.watch_chat`
- `notification.watch_tg_bot`

加载时会统一归一化到 `notification.watch`。

### 4.3 回退规则
如果没有独立 `watch` 目标，则回退到现有：

- `notification.admin_chat`
- `groups.admin_chat`
- `notification.tg_bot`

这样现有账号不需要一次性改完配置也能工作。

## 5. 命令设计

### 5.1 `watch`
查询当前账号的值守摘要，内容压缩成适合 Telegram 快速阅读的几行。

当前语义：

- 手动查询结果统一回复到 `admin_chat`
- 不再把手动查询正文转发到独立值守通道
- 主动值守播报仍然走 `watch chat / watch tg bot`

- 状态 / 模式 / fk 开关
- 当前预设 / 当前任务 / 当前策略 / 当前学习状态
- 资金 / 余额 / 总收益
- 24h 胜率 / 盈亏 / 回撤 / 样本量
- 当前盘面 / 温度 / 相似样本 / 当前建议
- 最近决策标签

### 5.2 `watch fleet`
查询多账号值守摘要：

- 每个账号一行
- 重点显示状态、任务、策略、学习状态、24h 盈亏/回撤、温度
- 优先把更需要关注的账号排前面

### 5.3 `watch learn`
查询学习链路值守摘要：

- 当前策略版本
- 候选总数 / 最新候选
- 最近评估结果
- 当前影子候选与样本表现
- 当前灰度候选
- 最近转正/回滚事件

### 5.4 `watch alerts`
查询告警摘要：

- 当前风险：看“现在就要处理什么”
- 最近播报：看“最近哪些问题在反复出现”

## 6. 主动播报设计

### 6.1 统一事件入口
新增统一的值守事件记录与发送链路：

- `tg_watch.record_watch_event(...)`
- `zq_multiuser._emit_watch_event(...)`

它负责两件事：

1. 把事件写入运行时 `watch_alerts / watch_event_state`
2. 按节流规则决定是否真正发到值守通道

### 6.2 当前已接入的主动事件
1. 任务包切换
2. 任务接管本轮
3. 模型预测超时
4. 入场风控暂停
5. 连输风控暂停
6. 影子验证通过恢复真钱下注
7. 影子验证未达标继续暂停
8. 资金不足暂停
9. 资金恢复 / 资金同步恢复
10. 学习影子开启/关闭
11. 学习灰度启动
12. 学习候选转正
13. 学习候选回滚

### 6.3 节流规则
每个事件记录：

- `event_type`
- `fingerprint`
- `last_sent_at / last_sent_ts`
- `last_seen_at / last_seen_ts`
- `count`

只有满足下面任一条件才会再次发送：

1. 指纹变化
2. 超过该事件的节流窗口
3. 明确指定 `throttle_sec = 0`

这能避免同一轮重复暂停、同一类资金不足或同一候选重复操作把值守通道刷满。

## 7. 告警摘要设计

### 7.1 当前风险
`watch alerts` 会实时检查当前账号/多账号的状态，优先列出仍在生效的问题，例如：

- 资金不足暂停
- 自动暂停仍在倒计时
- 学习灰度仍在运行
- 24h 偏冷且盈亏转负

### 7.2 最近播报
同时会读取最近累积的 `watch_alerts`，只展示：

- `warning`
- `error`

每一条包含：

- 最近通知时间
- 账号
- 事件标签
- 次数
- 首行摘要

这样可以快速识别“是一次性问题，还是短时间内反复出现的问题”。

## 10. 本阶段结论
`I. Telegram 值守播报 V1` 完成后，脚本具备了一个清晰的个人值守出口：

- 可以把摘要发到指定 TG 通道
- 关键事件会主动播报
- 同类事件不会短时间刷屏
- 可以在 Telegram 里直接看当前摘要、多账号摘要、学习摘要、告警摘要

这让后续阶段如果继续扩展，就不需要先补“怎么观察”，而是可以直接在已有值守通道上继续加能力。
