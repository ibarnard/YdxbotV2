# Admin 驾驶舱与分层播报 V1 技术说明书

## 1. 文档目的
这份说明书记载 `codex/risk-history-v1` 分支在 `M. Admin 驾驶舱与分层播报 V1` 阶段的设计和当前落地情况。

本阶段解决的不是“再多发一些消息”，而是：

> admin_chat 里如何形成一个稳定、低噪声、可复盘的个人驾驶舱，并把每盘、每手、每局、每轮、每日这些层级的消息拆清楚。

---

## 2. 阶段定位
到 `L. 复盘摘要与人工决策支撑 V1` 为止，系统已经具备：

- 结构化盘面/决策/风控/执行/结算资产
- 任务、任务包、策略版本、自学习
- Telegram 值守播报与点查
- 复盘摘要与人工动作建议

现在缺的不是数据，而是 `admin_chat` 里的业务表达层：

- 刷新式驾驶舱要看什么
- 独立弹出的过程卡如何分层
- `盘 / 手 / 局 / 轮 / 日` 这些口径怎样统一

---

## 3. 语义约定
为避免当前代码里的 `round/current_round/round_key` 与业务“局/轮”混淆，本阶段采用以下约定：

1. `盘`
- 一次原始开奖结果刷新
- 对应当前 `history` 增量与现有 `round_key`

2. `手`
- 一次真实下注动作，从下单到结算闭合

3. `局`
- 一个可复盘的统计区间
- 不等于原始盘
- 默认由任务完成、连输链闭合、自动暂停恢复后重新起段、手动切换策略、切日等边界闭合

4. `轮`
- 一个完整执行周期
- 优先等于 `task run`
- 无任务时等于一次从 `st/resume` 到 `pause/finish` 的人工运行段

5. `日`
- 自然日聚合

## 5. 总体设计

### 5.1 消息分层
我不建议把所有信息继续塞进一个超大仪表盘里，而是拆成：

1. `驾驶舱消息`
- 固定一条
- 刷新覆盖
- 负责当前态

2. `手卡`
- 每次真实下注/结算的独立事件卡
- 负责过程态

3. `局卡`
- 一个统计局闭合时发一次
- 负责区间复盘

4. `轮卡`
- 一个任务轮或人工轮结束时发一次
- 负责执行周期总结

5. `日卡`
- 当日聚合
- 负责自然日总结

6. `告警卡`
- 只发真正打断注意力的事

### 5.2 驾驶舱内容
驾驶舱 V1 重点展示：

- 账号、刷新时间、软件版本
- 当前脚本状态、模式、模型
- 近 40 盘结果
- 当前盘口：盘号、盘面、温度、尾部走势
- 当前判断：当前建议、限档、相似样本
- 当前手况：待结算/空仓、方向、金额、bet_id
- 上一手结算
- 当前局与当前轮
- 当前策略、动态档位、policy 版本、learn 状态
- 关键押注参数
- 资金与收益
- 24h 样本、胜率、盈亏、回撤、观望、阻断
- 风控开关与当前连输

## 6. 当前已落地
本次起步实现优先落了 `M1 + M2`：

### 6.1 M1 语义聚合读模型
当前先不引入新的复杂落库，而是优先从已有数据源聚合驾驶舱所需视图：

- `runtime`
- `history`
- `bet_sequence_log`
- `history_analysis.build_current_analysis_snapshot(...)`
- `history_analysis.build_policy_evidence_package(...)`
- 当前任务、任务包、策略版本、自学习中心

### 6.2 M2 驾驶舱 V1
当前已把 admin 驾驶舱接到：

- `status`
- `dashboard`
- 真实下单后自动刷新
- 结算后自动刷新

同时补上了更真实的 `当前押注状态` 展示：

- 手动暂停
- 已关闭
- 自动暂停（带剩余局数/原因）
- 运行中
- 已暂停
