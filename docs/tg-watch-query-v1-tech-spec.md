# Telegram 值守查询补强 V1 技术说明书

## 1. 文档目的
这份说明书记录 `codex/risk-history-v1` 分支在 `K. Telegram 值守查询补强 V1` 阶段新增的值守查询能力。

本阶段不是再做新通知体系，而是把已有值守出口补成“更快查、更少噪声、更适合个人日常盯盘”。

---

## 2. 阶段定位
到 `J. 执行稳定性与启动自检 V1` 为止，系统已经具备：

- 指定值守通道
- 主动播报关键事件
- 摘要与告警查询
- 启动自检与运行异常可见

当前还差一层“更细但仍然短”的查询能力：

- 风控我现在到底卡在哪里
- 任务和任务包当前推进到了哪里
- 资金和余额现在是什么状态
- 多账号下我想点名看某个账号，不想先切上下文
- 临时不想被主动播报刷屏，但手动查询仍然可用

---

## 3. 设计边界

### 3.1 本阶段完成
1. `watch risk / watch task / watch funds`
2. `watch <账号名|ID>`
3. `watch learn [账号名|ID]` 细节补强
4. `watch quiet [分钟|off]`
5. 对应测试、文档、回归

### 3.2 本阶段明确不做
1. 独立 Telegram 机器人命令体系重写
2. 复杂告警规则引擎
3. 定时报表平台
4. 运营后台
5. 自动停机、自动回滚、自动转正

---

## 4. 查询命令设计

### 4.1 `watch risk [账号名|ID]`
输出压缩后的风控视图：

- 当前状态
- `fk` 开关位
- 当前建议 / 限档
- 自动暂停、手动暂停、资金暂停
- 24h 盈亏 / 回撤 / 温度
- 最近阻断原因 / 影子验证状态

### 4.2 `watch task [账号名|ID]`
输出任务视图：

- 当前预设
- 当前任务包 / 包状态
- 当前任务 / 进度
- 触发模式
- 最近动作 / 原因 / 时间

### 4.3 `watch funds [账号名|ID]`
输出资金视图：

- 菠菜资金
- 账户余额
- 总收益 / 本轮收益
- 盈利目标 / 盈停
- 当前下注金额
- 余额状态
- 资金暂停状态

### 4.4 `watch <账号名|ID>`
直接把指定账号的总览回复到当前管理员 chat。

特点：

- 手动查询结果不再转发到独立值守通道
- 消息主体切换为被查询账号的摘要内容
- 不要求先切换当前账号上下文

### 4.5 `watch learn [账号名|ID]`
在原有学习摘要基础上补充：

- 候选 ID
- 影子候选 ID
- 灰度候选 ID
- 灰度目标账号

仍保持摘要格式，不展开成长报告。

## 5. 低噪声控制

### 5.1 `watch quiet`
新增轻量静音能力：

- `watch quiet 30`
- `watch quiet off`
- `watch quiet`

### 5.2 作用范围
静音只影响：

- 主动值守播报

不影响：

- 手动执行的 `watch *`
- `doctor`
- 普通管理员命令回显

### 5.3 当前实现
运行态新增：

- `watch_quiet_until`
- `watch_quiet_until_ts`

主动播报在真正发送前会检查静音状态；如静音仍生效，则：

- 事件照样记录
- 不实际发消息

这样既能降低噪声，又不会丢掉最近事件痕迹。

---

## 6. 值守摘要与告警变化

### 6.1 总览补充
`watch` 总览新增值守模式展示：

- 正常
- 静音剩余时间

### 6.2 告警补充
`watch alerts` 新增一类低优先级当前状态：

- 值守静音中

作用不是告警，而是告诉你“为什么这段时间没收到主动播报”。

## 9. 本阶段结论
`K. Telegram 值守查询补强 V1` 完成后，值守通道从“能看摘要”提升到“能快速点查关键切面”：

- 能查风控、任务、资金
- 能点名查某个账号
- 能更快看学习候选 ID 和灰度目标
- 能在需要时临时静音主动播报

这让 TG 值守更接近个人脚本真正的日常使用方式，而没有越界到复杂运营系统。
